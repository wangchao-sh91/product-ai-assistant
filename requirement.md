# AI Ops Knowledge Copilot Requirements

## 1. 项目背景

本项目定位为一个面向研发团队的 AI 应用示范项目，目标是构建一个“研发知识库与故障助手 Copilot”。项目既要体现传统后端工程能力，也要覆盖当前主流 AI 应用开发能力，适合作为个人转型 AI 应用开发的代表性项目写入简历。

项目将围绕以下真实研发场景展开：

- 研发知识问答
- 故障排查辅助
- Runbook/SOP 推荐
- 变更影响分析

## 2. 项目目标

### 2.1 业务目标

- 为研发、测试、运维人员提供统一的知识问答入口
- 缩短故障定位时间，辅助排查线上问题
- 沉淀企业内部知识资产，提升文档复用率
- 在回答中提供引用依据，降低“AI 幻觉”风险

### 2.2 个人项目目标

- 覆盖主流 RAG 技术链路
- 展示多 Agent 协作设计与工程落地能力
- 体现基于 Python 的完整后端系统设计、数据处理、评测与可观测性能力
- 形成可演示、可扩展、可写入简历的完整项目资产

### 2.3 技术目标

- 后端服务统一使用 Python 技术栈开发
- 优先采用单后端模块化架构，降低个人项目复杂度
- 保留后续按职责拆分 API 服务、异步任务和离线处理任务的演进空间

## 3. 用户角色

- 研发工程师：查询系统设计、接口说明、排障文档、历史事故
- 测试工程师：查询测试方案、已知缺陷、环境问题处理方式
- 运维/SRE：查询 Runbook、告警处理流程、历史工单和变更记录
- 团队负责人：查看知识沉淀效果、常见问题分布、使用统计

## 4. 核心功能范围

### 4.1 知识库管理

- 支持导入 Markdown、PDF、DOCX、TXT、HTML 等文档
- 支持导入 Postmortem、FAQ、Runbook、技术方案、接口文档
- 支持按系统、模块、环境、文档类型、更新时间维护元数据
- 支持增量更新、重建索引和文档版本管理

### 4.2 智能问答

- 支持自然语言问题输入
- 支持基于上下文的多轮问答
- 支持答案引用来源与证据片段展示
- 支持“无足够证据时拒答或降级回答”

### 4.3 故障助手

- 支持根据报错信息、告警描述、日志片段进行问题分析
- 支持结合知识库和历史事故进行相似问题匹配
- 支持生成排查步骤、优先级建议和可能根因列表
- 支持推荐对应 Runbook/SOP 文档

### 4.4 变更影响分析

- 输入需求变更、系统模块或服务名称
- 输出可能影响的上下游系统、接口、配置项和风险点
- 给出建议验证项与回归测试范围

### 4.5 多 Agent 协作

- 支持路由 Agent 识别任务类型
- 支持检索 Agent 负责查询改写、召回和重排
- 支持故障分析 Agent 负责日志/告警/历史工单综合分析
- 支持 Runbook Agent 负责生成可执行排查步骤
- 支持 Critic Agent 对答案的证据充分性进行复核

### 4.6 观测与评测

- 记录用户问题、检索结果、模型回答、引用文档
- 记录 Agent 调用链路、耗时、失败原因
- 支持构建评测集，对召回率、答案准确率、引用命中率进行评测

## 5. 非功能性要求

### 5.1 可用性

- 常规问答在 8 秒内返回首个完整结果
- 在外部依赖不可用时支持清晰降级提示
- 对无结果、低置信度结果进行显式标记

### 5.2 可扩展性

- 文档接入方式可扩展
- 大模型供应商可替换
- 向量库与重排模型可替换
- Agent 数量与能力可扩展

### 5.3 可维护性

- 模块边界清晰，支持独立部署
- 关键流程具备日志、Tracing 与错误追踪
- 提供配置化参数，如 chunk size、top k、rerank 开关

### 5.4 安全性

- 支持基础鉴权
- 支持知识库权限控制的预留设计
- 对敏感日志、工单字段进行脱敏

## 6. MVP 范围

首版建议聚焦在以下最小可用能力：

- 导入 3 类文档：Markdown、PDF、Runbook
- 实现基础 RAG：解析、切分、Embedding、向量检索、答案生成
- 实现 Hybrid Retrieval：向量检索 + 关键词检索
- 实现 Rerank 提升召回质量
- 实现 3 个 Agent：Router、Retriever、Incident Analyst
- 提供基于 Python 后端的 Web 聊天页或 Swagger API 演示入口
- 提供引用来源、检索证据和简单会话历史

## 7. 二期增强范围

- 接入工单系统、告警系统、CMDB、日志系统
- 增加 Runbook Agent 和 Critic Agent
- 支持多租户与权限隔离
- 支持知识图谱或服务依赖图辅助分析
- 支持离线评测、人工反馈闭环和 Prompt 版本管理

## 8. 典型用户故事

### 用户故事 1：研发知识问答

作为研发工程师，我希望输入“支付回调幂等是怎么设计的”，系统能返回规范答案，并附上设计文档和接口说明的引用片段。

### 用户故事 2：线上故障排查

作为值班工程师，我希望输入“订单服务 502，网关报 upstream timeout”，系统能结合历史事故和 Runbook 给出排查建议。

### 用户故事 3：变更影响分析

作为开发负责人，我希望输入“库存扣减逻辑新增异步重试”，系统能提示可能影响的下游系统、配置项和回归测试范围。

## 9. 验收标准

### 9.1 功能验收

- 成功导入至少 50 篇知识文档并建立索引
- 至少支持 20 条标准问答样例正确返回引用答案
- 至少支持 10 条故障场景样例输出可执行排查步骤
- 问答结果可展示引用来源、文档标题和证据片段

### 9.2 技术验收

- 具备完整 RAG 链路：解析、切分、检索、重排、生成
- 具备基本多 Agent 工作流
- 具备 Tracing、日志与基础评测脚本
- 关键配置可通过环境变量或配置文件调整

### 9.3 展示验收

- 可本地一键启动核心服务
- 可通过 UI 或 API 演示知识问答与故障分析
- 可输出项目架构图、流程图和评测结果截图

## 10. 成功指标

- Top 5 检索命中率 >= 80%
- 带引用答案覆盖率 >= 90%
- 故障场景人工主观可用率 >= 70%
- 核心问题平均响应时间 <= 8 秒

## 11. 约束与假设

- 项目为个人练习项目，优先实现工程完整性而非企业级复杂度
- 后端统一使用 Python，避免多语言切换带来的学习与维护成本
- 外部系统接入可先使用 Mock 数据替代
- 数据规模以本地单机部署可承受范围为主
- 首版不强制实现复杂权限体系和多租户隔离
